## 2\.2 分布式是什么？

一台主机的硬件资源是有上限的！！\!

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MTMwYmIwNzMzNzhmZjI3NjViZjE1MDYyNmFiODQxZGZfMWYwNmJiNGM1NjUzYjk3MGFhZTA1NmVjMjJjZTk2ODVfSUQ6NzY1NTUzMjM3NDg2NDc1OTc1Nl8xNzgyNTYyODA0OjE3ODI2NDkyMDRfVjM)

服务器每次收到一个请求，都是会消耗上述的一些资源的

如果同一时刻，处理的请求多了，此时就可能会导致某个硬件资源不够用了

无论是哪个方面不够用了，都可能会导致服务器处理请求的时间变成，甚至于处理出错\~

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZDRlMDU1OGRhYjc2YmE0NjBlNDM1MGMzNWY3MTZmYjBfNDM5YTk2YzdmNTVhMjM1OGEwMDk5MThjM2UwNzE1ODJfSUQ6NzY1NTUzMzU3NTAzNTA3OTY0NF8xNzgyNTYyODA1OjE3ODI2NDkyMDVfVjM)

如果我们真遇到了这样的服务器不够用的场景，怎么处理呢？

1. 开源 \-\- 简单粗暴，增加更多的硬件资源\~

2. 节流 \-\- 软件上优化（各凭本事了，需要通过性能测试，找到是哪个环节出现了瓶颈，再去对症下药）【这个是比较难的！对于我们程序员的水平要求就比较高了】

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZmY3MWM0ZWVmYmJiNWY2MTkxMmIxOWE4MWUzZjk1NzdfYmQxYWFhMWM0MzBhMGRlNTMyYTQyZWVmYTYzNmFiMTRfSUQ6NzY1NTUzNDMyNDk3NjI3NDM4OF8xNzgyNTYyODA0OjE3ODI2NDkyMDRfVjM)

如果一台主机扩展到极限了，但是还是不够用，就只能引入多台主机了 \-\- 不是说新的机器买来就可以直接解决问题了，需要软件上做出对应的调整和适配

一旦引入了多台追了，咱们的系统就可以称为是“分布式系统”

引入分布式，这是万不得已\~

## 2\.6 数据库分库分表

引入分布式系统，不光要能够去应对更高的请求量，并发量，同时也要应对更大的数据量

是否可能会出现，一台服务器已经存不下数据了呢？当然会存在。

虽然一个服务器，存储的数据量可以达到 几十个TB，即使如此也可能会存不下（比如短视频的业务场景）

**我们可以把数据库进行进一步的水平拆分**

**分库分表\~**

本来一个数据库服务器，这个数据库服务器上有多个数据库（指的是逻辑上的数据集合，create database 创建的那个东西）

**现在就可以引入多个数据库服务器，每个数据库服务器存储一个或者一部分数据库\~**

**如果某个表特别大，大到一台主机存不行，也可以针对表进行拆分**

具体怎么分库分表，如何实践还是要结合实际的业务场景来展开

## 2\.9 分布式系统小结

1. 单机架构（应用程序 \+ 数据库服务器）

2. 数据库和应用分离 \-\- 应用程序和数据库服务器，分别放到不同的主机上部署了

3. 引入负载均衡，应用服务器 \-\> 集群，通过负载均衡器，把请求比较均匀的分发给集群中的每个应用服务器（当集群中的某个主机挂了，其他的主机也可以承担服务，提高了整个系统的可用性，但还是最好快速去修复机器）

4. 引入读写分离（数据库主从结构），一个数据库节点作为主节点，其他N个数据库节点作为从节点，主节点负责写数据，从节点负责读数据，主节点需要把修改过的数据同步给从节点（我们MYSQL是有一套主从同步的机制的）

5. 引入缓存（冷热数据分离架构），进一步的提升了服务器针对请求的处理能力，遵循二八原则，Redis在一个分布式系统中，通常就扮演着缓存这样的角色\~ 引入的问题：数据库和缓存的数据一致性问题

6. 引入分库分表，数据库可以进一步扩展存储空间，针对数据量比较大的情况。具体的分法看实际业务需求就可以了。

7. 引入微服务，从业务上进一步拆分应用服务器，解决人的问题。从业务功能的角度，把应用服务器，拆分成更多的功能更单一，更简单，更小的服务器。

## 3\.1 redis 特性介绍

Redis 是一个内存中存储数据的中间件

1. 用于作为数据库，用于作为数据缓存

2. 在分布式系统中能够大展拳脚

**Redis的一些特性（优点），让我们的Redis应用的比较广泛**

**在内存中来存储数据，通过一系列数据结构**

- MySQL 主要是通过“表”的方式来存储组织数据的，“关系型数据库”

- Redis 主要是通过“键值对”的方式来存储组织数据的，“非关系型数据库”

**可编程的**：针对Redis的操作，可以直接通过简单的交互式命令进行操作，也可以通过脚本的方式，批量执行一些操作（可以带有一些逻辑，用的是 Lua）

Lua 也是一个编程语言

对应Redis最大的优势就是 **快**

1. Redis 数据在内存中，就比访问硬盘的数据库，要快很多

2. Redis 核心功能都是比较简单多的逻辑，做的事情没那么多，核心功能都是比较简单的操作内存的数据结构

3. 从网络角度上，Redis 使用了** IO 多路复用**的方式（epoll），使用一个线程，管理很多个 socket

4. Redis 使用的是单线程模型（虽然更高版本的Redis 引入了多线程，但是也是用在网络IO），这样的单线程模型，减少了不要的线程之间的竞争开销（多线程提高效率的前提是，CPU密集型的任务，使用多个线程可以充分的利用CPU多核资源）。但是 Redis 的核心任务，主要就是操作内存的数据结构，不会吃很多 CPU

还是那个道理，没有银弹

5. 个人是不太认可的，但是网上这么说的人很多，Redis 是使用 C语言开发的，所以就快（但是MySQL也是 C语言开发的啊，感觉有点牵强，但是对比使用其他语言也确实是有一定速度上的优势的）
