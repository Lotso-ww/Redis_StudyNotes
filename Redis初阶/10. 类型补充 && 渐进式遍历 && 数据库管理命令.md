## 1\.3 hyperloglog

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MWY0MTU3YjBjNjZmYmYwYTIyMTNjMjc5MjgwNDBhOTRfYzc2MzJiNjFhMTRlMmMzZWQxYmE4ZjQxNzUyNTJlZjFfSUQ6NzY2MTU0OTU5OTAzMTAwNDExMF8xNzgzODYxODQwOjE3ODM5NDgyNDBfVjM)

这个类型的应用场景只有一个，估算集合中的元素个数

Set 有一个应用场景，统计服务器的 UV（用户访问的次数）

我们使用 set 统计，肯定是可以统计 UV 的，但是最大的问题在于，如果 UV 数据量非常大，Set 会消耗很多的内存空间

假设 set 存储 userid 按照 8 个字节算

1亿 UV \-\> 8 亿字节 \-\> 0\.8 G \-\> 800 MB

Hyperloglog 可以最多使用 12kb 空间，实现上述效果

这个就很恐怖了

之所以要消耗这么大空间，是因为 set 要存储每个元素

而 hyperloglog 不存储元素的内容，但是能够记录 “元素的特征”，从而在新增元素的时候，能够知道当前新增的元素，是一个已经存在的元素，还是一个崭新的第一次出现的元素

用来计数（记录档期啊集合中有多少个不同的元素），但是不能告诉你这些元素都是啥

# 二\. 渐进式遍历 \-\- scan

Keys 一次性的把整个 redis 中所有的 key 都获取到

keys \*，这个操作比较危险，可能会一下得到太多的 key，阻塞 redis 服务器

通过渐进式遍历，就可以做到，即能够获取到所有的 key，同时又不会卡死服务器

不是一个命令，把所有的 key 都拿到，而是每执行一次命令，只获取其中的一部分，这样的话保证当前这一次操作不会太卡

要想得到所有的 key 就需要多次变量了，多次渐进式遍历命令

# 三\. 数据库管理命令

Mysql 中有一个重要的概念，叫做 database，一个 mysql 服务器上可以有很多个 database，一个 database 上可以有很多个表

在之前的学习中，感觉 redis 好像上来就是 key value

其实 redis 也是有 database 这样的概念的，只不过不像 mysql 那样随意

Mysql 上可以随心所欲的 创建/删除 数据库

Redis 中的 database 是现成的，咱们用户不能创建新的数据库，也不能删除已有数据库
