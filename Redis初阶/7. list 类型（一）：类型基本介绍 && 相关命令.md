1. 列表中的元素是有序的

“有序” 的含义，要根据上下文区分

有的时候，谈到有序，指的是 “升序” 和 “降序”

**有的时候，谈到的有序，指的是，顺序很关键。这里就是这种**

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZWEzZWZmZmVjYTU0MjcyN2M5ZWZiYmEyYTQ4OWEwZTZfY2UzMmZlNTcwYzBhMGY3MjkxN2Q4YjcxODNmYTFhMWZfSUQ6NzY1ODUxMTIyNTA4MDQ2NjY1NF8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

同样一个词，怎么理解，务必要结合上下文，结合具体场景

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=YzY5N2M3ODVhOTVhYTZjZTIxMjE3YTRlNTkyM2M2NjBfYTRjYzNjYWVhMzJkMzE4YTZkMjM0Y2FlNmY0ZWY0NjFfSUQ6NzY1ODUxMTYzMTYyMTg4NDg5Nl8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

2. 区分获取元素和删除元素的区别，主要就是lindex能获取元素的值，lrem也会返回被删除元素的值

3. 列表中的元素是允许重复的，像 hash 这样的类型，field 是不能重复的

4. 因为当前的 list，头和尾部都能高效的插入和删除元素，就可以把 List 当做一个 栈/队列 来使用了，队列又可以用到实现生产者消费者模型

Redis 有一个典型的应用场景，作为消息队列，最早的时候，就是通过 List 类型

后来 redis 又提供了一个 stream 类型

# 二\. List 的命令

## 2\.1 lpush \&\& lrange

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MjQxNzA3ZGIxZjQwOWI5NjJhODk1MDBmZTg3MDM4MmVfNTU3MTg3MjMyZDZiNTNiOWFlYzM5NGRjNDk4NjhhZmNfSUQ6NzY1ODUyMjc2NDM3NDA2ODQ1NF8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

**一次可以插入一个元素，一次也可以插入多个元素**。

一次插入多个，可以减少网络开销

按照顺序依次头插元素

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MjYxNzljZTM2Nzc4ZjQ0NWUzMzNiYzQxMDFjZGY5YTdfOTBiYzFmMWUxZjNhMDZkY2IwODJmMzIyNDNiNzg3ZmNfSUQ6NzY1ODUyMjk5MDYzNjA5MjYwNF8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

时间复杂度是 O\(1\)，这个就不需要多说了

返回值是 list 的长度

如果 key 已经存在，并且 key 对应的 value 类型，不是 list，此时 lpush 就会报错，redis 中所有的这种各种数据类型的操作，都是类似的规则

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NDRkYzczZjc5ZmM3YTgyMDAyYjQxOTlhZmEyY2UxOWJfNzAzNDkyNmU3YmUwOWFlNTM5OGQzZDc3OGM3OGQxMTJfSUQ6NzY1ODUyNDM1NTIxNDc1NzA2MV8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

谈到下标，我们往往会关注 超出范围 的情况

C\+\+ 中，下标超出范围，一般会认为这是一个 “未定义行为”  \-\> 可能会导致程序崩溃，也可能会得到一个不合法的数据，还可能会得到一个看起来合法，但是错误的数据，也有可能恰好得到一个符号要求的数据\.\.\.

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NjIyNjdjNDNiY2JiNTY0MDFjNDEwZjFjMGQ1ZGE0MTRfMWI4NmZkNDU5ODkyOWIxNjczMzI1ZjI1YTdkYTY5MzBfSUQ6NzY1ODUyNjUxMjcyMzcyNTI3MF8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

在 redis 中并没有采取上述的这两种措施

**Redis 的做法，是直接尽可能的获取到给定区间的元素，如果给定区间非法，比如超出下标，就会尽可能的获取对应的内容**

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=M2ZmODFkZDljYWE4YzY5M2U4ZmUyNDI0YzMzNjEzOTZfMjU1ZTg4MTM5ZWI1OTU3NjIxYWIzNGRkOTcyZDEyYTFfSUQ6NzY1ODUyNjkzOTA4MjEzMjcxMl8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

我们获取到了能获取到的元素

此处对于越界下标的处理方式，更接近于 Python 的处理方式（Python的切片）“鲁棒性”比较好（你对我越粗鲁，我就表现的越棒），程序的容错力更强

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NzExOGNhYWFkM2MxYjgzMTlhYTM2N2ZmY2YzZDZhYTVfYzdkMzA0ZjQwNGZjNTIzNDg5NmY5Y2RlYWE5MTM2ZDNfSUQ6NzY1ODUyODc2ODA5MjMxMDUxNV8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

是机器执行的快重要还是我们代码写的快重要，当然是我们自己写的快重要啊，机器快还是慢，那你升级设备呗

## 2\.7 blpop \&\& brpop

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZTNiOGFkNDk2ODNjMDA3NjIxODAxNTk3ZTUzYjQzZjVfOGU1MDljNTY4ZTdlODg3OGU3MWZkZTdkY2IyZDFmNzdfSUQ6NzY1ODU4NDY2NDQ3NDUzNjkyNF8xNzgzMTU1NTA4OjE3ODMyNDE5MDhfVjM)

阻塞：当前的线程不走了，代码不继续执行了

会在满足一定的条件之后，被唤醒

重点就是blpop和brpop

b就是 block

阻塞队列（BlockingQueue）\-\- 我们Linux中讲生产者消费者模型讲过这个

使用队列作为中间的“交易场所”

期望这个队列有两个特性：

1. 线程安全

2. 阻塞功能

    - 如果队列为空，尝试出队列，就会产生阻塞，直到队列不为空，阻塞解除

    - 如果队列为满，尝试入队列，就会产生阻塞，直到队列不满，阻塞解除

Redis 中的 list 也相当于 阻塞队列 一样

线程安全是通过单线程模型支持的
