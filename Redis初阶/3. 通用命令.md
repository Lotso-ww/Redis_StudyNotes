Redis 是一个客户端\-服务器 结构的程序

客户端和服务器之间通过网络来通信！

**分开的写法会产生更多轮次的网络通信，效率比较低，成本比较高（和直接操作内存相比）**

接收方收到一个数据，这个数据就要从物理层，到应用层层层分用，把每一层协议中的报头或者尾给拆掉，收到一个快递，要拆快递，要拆很多层

网卡是 IO 设备

更何况，你的客户端和服务器不一定在一个主机上，中间可能隔着很远

Redis 自身也是非常清楚上述问题，redis 的很多命令都是支持一次就能操作多个key的操作的

总之还是那句话，我们说的快慢还是要有参照物的。

**时间复杂度：O\(1\)**，有几个key就是O几

**返回值：删除掉的key的个数**

我们之前学MySQL的时候，我们强调，删除类的操作，drop database，drop table，drop from\.\.\. 都是非常危险的操作，一旦删除了之后，数据就没了

Redis 最主要的应用场景，就是作为缓存，此时redis 里存的只是一个热点数据，全量数据是在MySQL数据库中，此时redis中的key删除了几个问题不大，大不了重新从MySQL读就行了

**但是，当然如果把所有数据或者一大半数据一下都干没了，这种影响会很大，本来redis 是帮 MySQL 负重前行，redis 没数据了，大部分的请求就直接打给MySQL，然后就容易把MySQL搞挂掉**

但是归根结底，还是不要乱删数据

# 四\. Redis 的 key 过期策略（经典面试题）

Redis 的 key 的过期策略是怎么实现的，一个redis中可能同时存在很多很多key，这些 key 中可能有很大一部分都有过期时间，此时，redis 服务器咋知道那些 key 已经过期要被删除，哪些 key 还没过期

如果直接遍历所有的 key，显然是行不通的，效率非常低

Redis 整体的策略是：

1. 定期删除

2. 惰性删除

两种结合使用

惰性删除，假设这个 key 已经到过期时间了，但是暂时还没删除它，紧接着，后面又一次访问，正好用到了这个 key ，于是这次访问就会让 redis 服务器触发删除 key 的操作，同时再返回一个 nil

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZTI4YjAyODUzYWIwNTcwYTM5YTg2YjU4ZjgxMmNkYjZfNWQxMGYzMTg3MDM4YTVmZGQ3N2EzYmEyZGJlZWZkMjlfSUQ6NzY1NjYzNjA0NDE2Nzg0MzAzMF8xNzgyNzIwMjg1OjE3ODI4MDY2ODVfVjM)

但是只靠惰性删除是不靠谱的，此处也需要结合定期删除的操作，每次抽取一部分进行验证过期时间，保证这个抽取检查的过程，足够快

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZDljM2FmNDEwOTg4ZGQxN2QyMmNhZmJiNWRkNDJjOWZfM2M5ZjdlYzVhYmEyZjZmN2EzMzEzYmIzMGU2OTYyMTdfSUQ6NzY1NjYzNjY4NDc1MDE5NTkzM18xNzgyNzIwMjg2OjE3ODI4MDY2ODZfVjM)

为什么对于定期删除的时间有明确的要求呢？

因为 redis 是单线程的程序，主要的任务（处理每个命令的任务，刚才扫描过期 key\.\.\.）

如果扫描过期 key 消耗的时间太多了，就可能导致正常处理请求命令就被阻塞了（产生了类似于执行 keys \* 的效果）



即使虽然有了上面两种策略结合，整体的效果一般，仍然可能会有很多过期的 key 被残留了，没有及时删除掉

redis 为了对上述进行补充，还提供了一系列的内存淘汰策略！

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MDI5NjViMGU3MDNlNTU3ZTIxOTAyNjM5ZDNkYjYzNTdfOTYzNThlNzNmYTA5YTkwZTNjZjc4M2EwOGJhZDViMzFfSUQ6NzY1NjYzNzgwODc3ODI1MTQ0OV8xNzgyNzIwMjg1OjE3ODI4MDY2ODVfVjM)

为啥 redis 没有采取这种定时器的方式呢？这个事情，我们很难考证为啥，个人感觉，基于定时器的实现，势必是要引入多线程了，redis 早期版本就是奠定了单线程的基调，引入多线程就被打破了作者的初衷

# 五\. 定时器的实现原理

**定时器：在某个时间到达之后，执行指定的任务**

1. 基于优先级队列/堆来实现

正常的队列是先进先出，优先级队列则是按照指定的优先级，先出

**啥叫优先级高，这个是可以自定义的，在 redis 过期 key 的场景中，就可以通过 “过期时间越早，就是优先级越高”**

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MWIwNDAwYjVhNWE1ZWNhNzg5YzE1MzU0MTU3MmUyYjBfOWEwZmRiYzA2MTg5NzE1MGYyNjZlNDFmZjUwZGRlMWNfSUQ6NzY1NjcxNzE4Mzg1MDY3OTUzOV8xNzgyNzIwMjg1OjE3ODI4MDY2ODVfVjM)

现在假定有很多 key 设置了过期时间

就可以把这些 key 加入到一个优先级队列中，指定优先级规则是过期时间早的，先出队列

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NTk2MTMzODMyMmY3YWQ0MTRlYTNiN2QzY2FmNjNiZjJfYmY4OWQ5Yjc4YTYyNGE4MTEwMjc5MWNjN2FiYmUyOGZfSUQ6NzY1NjcxNzI4MDA3MTQ3MDAxMl8xNzgyNzIwMjg2OjE3ODI4MDY2ODZfVjM)

**此时 扫描线程，不需要遍历所有 key 只盯住这一个队首元素即可**

另外在扫描线程检查队首元素过期时间的时候，也不能检查的太频繁

此时做法就是可以根据当前时刻和队首元素的过期时间，设置一个等待

当时间差不多到了，系统再唤醒这个线程

**此时 扫描线程，不需要高频扫描队首元素，把 CPU 的开销也节省下来了**

万一在线程休眠的时候，来了一个新的任务，是11:30要执行

**我们可以在新任务添加的时候，唤醒一下刚才的线程，重新检查一下队首元素，再根据时间差距重新调整阻塞时间即可**

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NDk4MDE0MWQ0Y2U4ZWJjYWM3ZjgzOTY4ZWYxYjFiM2RfNzU4NzZiODkyZGI1NzcyODM4YjQzMzEzNzJjMGQwY2JfSUQ6NzY1NjcxODM3MTk3MTI3MTg4M18xNzgyNzIwMjg2OjE3ODI4MDY2ODZfVjM)



2. 基于时间轮实现的定时器

我们把时间划分成很多小端（划分的粒度，看实际的需求）

我们把均匀的小段分在圆环上，这个时间轮是怎么玩的呢，一个指针指向其中一个小段，照着一下方向去走

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=OTE4MTVmMmIwMDU5NDlkYTUzNzRlYjY1NGFjZTZlOTVfMzJjNGVlNDBjMzYwNzdmMWVkNjFjYjJmM2YxMGNkYTZfSUQ6NzY1NjcxODgwNzY0MTA2NjcyOF8xNzgyNzIwMjg2OjE3ODI4MDY2ODZfVjM)

每个小段上都挂着一个链表，每个链表都代表一个要执行的任务（相当于一个函数指针，以及对应的参数啥的，Java中也可以通过对象来实现类似的效果）

**假设需要添加一个 key，这个 key 在 300ms 之后过期**

此时这个指针，就会每隔固定的间隔（此处约定是100ms）

每次走到一个格子，就会把这个格子上链表的任务尝试执行一下

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NzEzNWFmMjQ2ZGMxODI5N2Y3ZDI0NjYzZGJiYzVmOWRfNWI4YzE2NDRkMDJhYTJiOGUzNDk3MGQ4MTBhMDM3NDNfSUQ6NzY1NjcxOTgyNDU5NzY2NzAzNl8xNzgyNzIwMjg2OjE3ODI4MDY2ODZfVjM)

对应时间轮来说，每个格子是多少时间，一共多少个格子，都是需要根据实际场景，灵活调配的



**此处大家一定要注意，redis 并没有采取上述的方案，但是我们还是了解一下这两个方案，都是属于高效的定时器的实现方式，很多场景可能都会用到**

**在 redis 源码中，有一个比较核心的机制，是 事件循环，和这个机制有点类似，但是又不太一样**
