# 二\. String 类型的应用场景

## 2\.1 缓存

String 类型应用场景

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZDk0OTYwODIwY2Y4MDA2MWQ3MzE0Y2Y1ZDIwOWE5NzlfOWU4Yjk3MjFmNzMwZTczZWY5ODZjNDU2MzhhNjQ1ODZfSUQ6NzY1Nzc5MzUzMDk4NzQ2NTY4N18xNzgyOTg4NDQ2OjE3ODMwNzQ4NDZfVjM)

整体的思路：应用服务器访问数据的时候，先查询 redis，如果 redis 上数据存在了，就直接 redis 取数据交给应用服务器，不继续访问数据库了

如果 redis 上数据不存在，再读取 MySQL，把读到的结果，返回给应用服务器，同时，把这个数据也写入到 redis 中

Redis 这样的缓存，经常用来存储 “热点” 数据 \-\- 热点数据就是高频被使用的数据，这个定义方式，结合业务场景有很多种方式。

刚才上述描述的过程，就是相当于把最近使用的数据作为热点数据（暗含了一层假设，某个数据一旦被用到了，那么很可能在最近这段时间被反复用到）

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MDBiOTY5NDlhOTM5ZGUzN2JhNThjY2EzNmJkZDM1ZjRfYjU0YjdmMzUxM2ZiMTUyMGQ1ODljNTY0ZjI1NzcyZTlfSUQ6NzY1Nzc5NDY2NjY3MDUyNTYyMF8xNzgyOTg4NDQ2OjE3ODMwNzQ4NDZfVjM)

上述策略，存在一个明显的问题：随着时间的推移，肯定会有越来越多的 key 在 redis 上访问不到，从而从 mysql 读取并写入 redis 了

此时 redis 中的数据不是就越来越多嘛？

1. 在把数据写到 redis 的同时，给这个 key 设置一个 **过期时间**

2. Redis 也在内存不足的时候，提供了淘汰策略，后面再细说

我是一个病人，相当于一个客户端

医生要给我诊病，医生给我提供服务，相当于服务器

由于病人需要多次访问服务器，服务器这边就需要很好的了解到病人每次复查时候的状态\~

但是医院有很多个医生，也就是多个服务器，这些服务器，是以 负载均衡 的方式来提供服务的

同一个病人，两次来医院看病，遇到的医生可能是不同的

此时，如果每个医生都只是靠自己的记忆，来存储病人的状态，此时两次访问到不同的医生，医生就很难知道病人之前的情况了

医院正确的做法，是使用一套系统，记录病人的病例，诊断结果，治疗情况\.\.\.，让多个医生共享（这个就是所谓的会话，客户端和服务器在交互过程中产生的一些专属于该客户端的中间状态的数据）

如果每个应用服务器，维护自己的会话数据，此时彼此之间不共享，用户请求访问到不同的服务器上，就可能会出现一些不正确处理的情况了\~

统一存在redis中

**此时所有的会话数据（病人的信息）就被各个服务器（医生）共享了**

# 三\. 什么是业务

业务其实就是一个公司/一个产品，是如何解决一个/一些系列问题的

**解决问题的过程，就可以称为是 “业务”**
