# 二\. 认识 RESP

为什么我们能编写出一个自定义的 redis 客户端？

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=ZTE2MDZhMDVhMTQ3MjEzYWM2ZWU0MjgzZDRjY2FhZDFfOWI5YWQ5ZWViMDY2NTVjNTg0Y2Y1ZjQxMTBkMDBlOTZfSUQ6NzY2MTkwODM2Njk0NzU2ODYwMl8xNzgzOTI4Njg1OjE3ODQwMTUwODVfVjM)

客户端可以按照这里的应用层协议，发送请求

服务器按照这个协议进行解析，再按照这个协议构造响应

客户端再解析这个响应

之所以上述通信能够完成，是因为开发客户端的人和开发服务器的人都清楚的知道 协议 的细节

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=OWNmZWM1MzIyMDNjMDJlYTgyYzc0ZDJjMDk0Y2FmYmVfYTJjNzgxYTExNmUzOWRkZjA1NjUzZjNjY2Q0YTc2MjVfSUQ6NzY2MTkwODcxNzU3ODk4MDMxMV8xNzgzOTI4Njg1OjE3ODQwMTUwODVfVjM)

咱们作为第三方，要想开发一个 redis 客户端，也就需要知道 redis 的应用层协议

而且我们也可以知道这个应用层协议，官方文档就可以看到

虽然 腾讯 没公开 qq 的协议，但是我看到说网上还是有一些开源项目，实现了自定制的 qq 客户端呀

可以通过一些抓包/逆向手段，猜测 qq 的应用层协议是啥样的，这个过程全靠猜，猜多猜少，猜对猜错，取决于水平\+运气，这个逆向工程是很难的



**RESP 协议的优点：**

1. 简单好实现

2. 快速进行解析

3. 肉眼可读

4. 传输层这里基于 TCP，但是和 TCP 又没有强耦合

5. 请求和响应之间的通信模型是一问一答的形式 \-\- 客户端给服务器发一个请求，服务器返回一个响应



客户端给服务器发送的是 redis 命令（bulk string 数组的形式发送的）

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=NmU5NzllNjYwZDlkOGYyNGUwMmQzODk2MDcxMjIzZWVfMmFkZmJlNmNiMjkzNzEyNjNlNTRmNGI1ODA4MTQ1Y2VfSUQ6NzY2MTkxMjE1MDA1NTQ1NjAxMl8xNzgzOTI4Njg1OjE3ODQwMTUwODVfVjM)

上面是 RESP 最核心的说明，RESP 我们讲这个主要是希望大家可以了解 redis 底层的通信原理，即使不了解，其实也没关系，不影响后续代码的编写和学习



**格式符号**

![Image](https://internal-api-drive-stream.feishu.cn/space/api/box/stream/download/authcode/?code=MGIxM2E4YjE4YjA2ZDIyMTEzNmYwMjY3MTE4YWQ5YTVfNjFkZWNiMTJjZmE1NWEyMDNmN2Y5ZGVmMGJmZTdlN2FfSUQ6NzY2MTkxMzcxNjQxMDA1OTczMl8xNzgzOTI4Njg1OjE3ODQwMTUwODVfVjM)

**Simple string**** 只能用来传输文本**

**Bulk string**** 可以传输二进制数据 \-\- 比如命令就是用的这种形式**

因此我们的 redis 客户端服务器要做的工作就是

1. 按照上述格式，构造出字符串喜，往 socket 中写入

2. 从 socket 中读取字符串，按照上述格式解析



我们写代码真的要按照上述的协议，解析/构造字符串嘛

不用，这套协议，公开很久了，已经有很多大佬，实现了这套协议的解析/构造，只要使用这些大佬们提供的库，就可以比较简单方便的来完成和 redis 服务器通信的操作了
